# 需求文档

## 简介

本文档概述了重建C盘瘦身工具的需求 - 一个安全且智能的磁盘空间管理应用程序。该工具将文件和目录从C盘移动到其他驱动器，同时通过符号链接维护系统功能，具备全面的安全机制、回退功能和多语言支持。

主要目标是创建一个强大、用户友好的应用程序，能够安全地将文件从系统驱动器重新定位以释放空间，而不会破坏系统功能或用户应用程序。

## 需求

### 需求1：安全文件迁移算法

**用户故事：** 作为系统管理员，我希望有一个安全的算法在迁移前分析文件依赖关系，以便在不破坏系统功能的情况下移动文件。

#### 验收标准

1. 当系统分析目录进行迁移时，系统应识别所有系统关键文件和依赖关系
2. 当文件被识别为系统关键文件时，系统应将其标记为不可移动并通知用户
3. 当分析文件依赖关系时，系统应检查注册表项、快捷方式和应用程序配置
4. 如果文件有活动句柄或正在使用中，系统应推迟操作并通知用户
5. 当创建迁移计划时，系统应验证所有依赖关系都可以通过符号链接维护

### 需求2：全面回退系统

**用户故事：** 作为用户，我希望能够完全回退任何迁移操作，以便在出现问题时将系统恢复到原始状态。

#### 验收标准

1. 当开始任何迁移操作时，系统应创建完整的备份清单
2. 当创建备份清单时，应包括文件路径、权限、时间戳和注册表项
3. 当请求回退时，系统应将所有文件恢复到原始位置
4. 当执行回退时，系统应删除所有创建的符号链接
5. 如果回退部分失败，系统应提供详细的错误报告和手动恢复说明
6. 当回退完成时，系统应验证系统完整性并报告成功状态

### 需求3：系统重启和注销检测

**用户故事：** 作为用户，我希望系统检测何时操作需要重启或注销，以便我可以相应地计划并维护系统稳定性。

#### 验收标准

1. 当分析文件进行迁移时，系统应识别需要系统重启才能移动的文件
2. 当系统文件或运行中的服务受到影响时，系统应要求用户确认重启
3. 当需要注销时，系统应通知用户并提供安排操作的选项
4. 如果需要系统重启，系统应创建重启后验证任务
5. 当系统在迁移后重启时，系统应自动验证所有符号链接是否正常工作
6. 如果重启后验证失败，系统应自动启动回退程序

### 需求4：多语言支持

**用户故事：** 作为国际用户，我希望应用程序支持多种语言，以便我可以使用我的首选语言。

#### 验收标准

1. 当应用程序启动时，应检测系统语言并使用适当的本地化
2. 当用户选择不同语言时，系统应立即更新所有UI元素
3. 当显示错误消息时，应以所选语言显示
4. 当生成报告或日志时，应包含本地化描述
5. 如果缺少翻译，系统应回退到英语并记录缺少的翻译

### 需求5：智能文件分析

**用户故事：** 作为用户，我希望系统智能分析哪些文件可以安全移动，这样我就不必手动确定文件安全性。

#### 验收标准

1. 当扫描目录时，系统应按安全级别对文件进行分类（安全、谨慎、不安全）
2. 当文件被分类为"不安全"时，不应可用于迁移
3. 当文件被分类为"谨慎"时，系统应要求明确的用户确认
4. 当分析应用程序时，系统应检查它们是否支持符号链接
5. 如果应用程序不支持符号链接，系统应推荐替代解决方案

### 需求6：进度监控和报告

**用户故事：** 作为用户，我希望在操作期间获得详细的进度信息，以便我可以监控过程并识别任何问题。

#### 验收标准

1. 当执行任何操作时，系统应显示带有文件名的实时进度
2. 当操作完成时，系统应生成详细报告
3. 当发生错误时，应记录带有时间戳和详细错误描述的日志
4. 当操作正在进行时，用户应能够安全地暂停或取消
5. 如果操作被取消，系统应自动回退任何部分更改

### 需求7：磁盘空间管理

**用户故事：** 作为用户，我希望看到磁盘空间分析和预测，以便我可以就文件迁移做出明智的决定。

#### 验收标准

1. 当分析目录时，系统应计算并显示潜在的空间节省
2. 当选择文件进行迁移时，系统应显示前后磁盘使用预测
3. 当选择目标驱动器时，系统应验证存在足够的可用空间
4. 如果空间不足，系统应阻止操作并建议替代方案
5. 当迁移完成时，系统应显示实际实现的空间节省

### 需求8：系统集成安全

**用户故事：** 作为系统管理员，我希望确保工具不会干扰系统操作，以便我可以在生产系统上安全使用它。

#### 验收标准

1. 当应用程序启动时，应检查管理员权限
2. 当检测到系统文件时，系统应应用额外的安全措施
3. 当Windows更新待处理时，系统应警告潜在冲突
4. 如果检测到杀毒软件，系统应检查兼容性
5. 当创建符号链接时，系统应验证NTFS权限得到保留

### 需求9：配置和首选项

**用户故事：** 作为用户，我希望配置工具的行为并保存我的首选项，以便我可以根据我的特定需求进行自定义。

#### 验收标准

1. 当首次启动时，系统应引导用户完成初始配置
2. 当用户更改设置时，应在应用程序重启后保持
3. 当创建迁移规则时，用户应能够将其保存为模板
4. 如果默认设置不安全，系统应使用保守的默认值
5. 当导入/导出设置时，系统应验证配置完整性

### 需求10：紧急恢复

**用户故事：** 作为用户，我希望在应用程序失败时有紧急恢复选项，以便即使工具不工作我也能恢复我的系统。

#### 验收标准

1. 当任何操作开始时，系统应创建紧急恢复脚本
2. 当应用程序崩溃时，恢复脚本应自动执行
3. 当迁移后系统启动失败时，系统应提供安全模式恢复选项
4. 如果符号链接损坏，系统应自动检测并修复它们
5. 当需要紧急恢复时，系统应提供逐步手动说明

### 需求11：开发者打赏系统

**用户故事：** 作为开发者，我希望集成安全的打赏机制，以便用户可以支持软件开发并获得收入。

#### 验收标准

1. 当应用程序启动时，系统应显示打赏页面选项卡
2. 当用户访问打赏页面时，系统应显示多种支付方式（微信、支付宝、BTC、USDT）
3. 当显示收款信息时，系统应从加密数据库中安全加载收款地址和二维码
4. 如果收款信息被篡改，系统应自动使用备用地址并记录安全事件
5. 当用户点击复制按钮时，系统应将收款地址复制到剪贴板并显示确认消息

### 需求12：打赏账号防篡改保护

**用户故事：** 作为开发者，我希望打赏账号信息受到保护，以防止被恶意篡改或替换。

#### 验收标准

1. 当存储收款地址时，系统应使用AES-256-CBC加密算法进行加密
2. 当验证收款地址时，系统应使用HMAC-SHA256进行完整性校验
3. 如果检测到地址被篡改，系统应立即恢复为硬编码的备用地址
4. 当加载二维码图片时，系统应验证图片完整性
5. 如果数据库被删除或损坏，系统应自动重建并使用备用收款信息

### 需求13：程序保护与防篡改

**用户故事：** 作为开发者，我希望程序本身受到保护，以防止被恶意修改或破解。

#### 验收标准

1. 当程序启动时，系统应验证主程序文件的SHA-256哈希值
2. 当检测到程序文件被修改时，系统应显示安全警告并拒绝运行
3. 当存储敏感配置时，系统应使用动态密钥进行加密
4. 如果配置文件被篡改，系统应使用默认安全配置并记录事件
5. 当程序运行时，系统应定期验证关键组件的完整性

### 需求14：机器码生成与验证

**用户故事：** 作为用户，我希望看到唯一的机器码，以便在需要技术支持时提供给开发者。

#### 验收标准

1. 当程序启动时，系统应基于硬件信息生成唯一机器码
2. 当显示机器码时，系统应使用易读的格式（XXXX-XXXX-XXXX-XXXX）
3. 当用户点击机器码时，系统应将其复制到剪贴板
4. 如果硬件信息获取失败，系统应使用备用方案生成机器码
5. 当机器码用于验证时，系统应确保其唯一性和稳定性
