# 乱码避坑指南（Delphi + dcc64）

本项目针对 Win64 编译（dcc64）统一做了编码与字符串修复。为避免再次出现“中文乱码/字符串未闭合(E2052)/语法错位(E2029)”等问题，请遵循以下实践。

## 一、统一编码策略
- 全部源文件（.pas/.dfm/.dpr/.dpk/.inc）统一保存为 UTF-8 with BOM。
- 不要在代码中使用 `{$CODEPAGE UTF8}` 等 Codepage 编译指令（dcc64 不支持/不可靠）。
- 若第三方代码含有 `{$CODEPAGE ...}`，应删除指令并将文件转换为 UTF-8 with BOM。

## 二、中文字符串与日志
- Pascal 源码中的中文字符串允许存在，但推荐：
  - 避免在核心算法/底层单元中使用大量中文字符串，以降低跨地区构建差异。
  - 运行日志与用户提示区分：
    - 日志（Log）：尽量用英文 ASCII（稳定、无乱码风险）。
    - UI 提示：可中文，但务必保证文件编码正确、字符串闭合完整。
- 常见错误：
  - 未闭合字符串 E2052 绝大多数由文件编码错位或字符串中夹入非法字符导致。
  - 解决办法：转换文件为 UTF-8 with BOM，替换可疑字符串为纯 ASCII 或重新手工录入中文。

## 三、DFM 与字体设置
- DFM 也应保存为 UTF-8 with BOM。若 DFM 中中文 Caption 显示为乱码，可：
  - 在 Form/Frame 创建时设置控件字体：
    - `Font.Name := 'Segoe UI'`（或 `Microsoft YaHei UI`）；
    - `Font.Charset := DEFAULT_CHARSET`。
  - 运行时再覆盖 Caption，避免设计期字符集干扰。

## 四、右键菜单/界面中文
- 若右键菜单或控件中文仍乱码，多半是控件字体/字符集或者源文件编码问题：
  - 检查该单元是否 UTF-8 with BOM；
  - 检查该窗体是否在 `FormCreate/AfterConstruction` 中统一设置字体；
  - 若是虚拟列表或自绘，需要 DrawText 时选择支持 Unicode 的 API（Delphi 新版默认 OK）。

## 五、uBasicProtection 与加密
- 已统一去除 `{$CODEPAGE UTF8}`，并修复了 Win64 不支持的“内联 var”声明写法。
- Key 派生统一为 `APassword`，兼容 `@2241114`。
- 文件/字节哈希使用逐块 `THashSHA2.Update` 计算，避免不兼容的重载。

## 六、接口声明被注释打断的风险
- 由于历史乱码，出现过“注释打断声明”，导致接口区的字段/方法不可见：
  - 例如：`EnableLogging`、`EncryptionType`、`Initialize` 等标识符报未声明。
  - 解决：逐行检查 `type` 区域，把字段/方法各自独立成行，注释放在行尾或上一行。

## 七、排错清单（Checklist）
- 编译时报：
  - E2052 未闭合字符串：
    - 定位行，逐字符检查是否存在乱码字符；
    - 将字符串替换为 ASCII 试编译；
    - 确保文件是 UTF-8 with BOM；
  - E2029 语法错位：
    - 检查本行的前几行是否有未闭合字符串或被乱码破坏的分号/括号；
  - E2003 未声明标识符：
    - 检查接口区声明是否被注释打断；
    - 检查 uses 顺序与依赖单元是否能编译；
  - F2063 无法编译使用的单元：
    - 优先修复最先抛错的那个单元，通常是字符串未闭合导致后续级联错误。

## 八、工具与建议
- 批量转换编码：
  - 建议使用 IDE 或脚本批量转换为 UTF-8 with BOM，并做时间戳备份（本项目已在根目录 `.backup_encoding_yyyyMMdd_HHmmss/` 备份）。
- 日志系统：
  - 若日志未生成，优先检查路径/权限；
  - 使用 `TStreamWriter.Create(path, True, TEncoding.UTF8)` 确保写入为 UTF-8。

## 九、建议的国际化方案（可选）
- 使用 `resourcestring` 定义字符串资源，或外置 JSON/INI 资源文件，运行时按语言加载；
- 中文资源置于专门文件，便于统一审校与查找。
